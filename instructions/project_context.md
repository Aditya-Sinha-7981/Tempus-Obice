**Version:** 1.0

# Purpose

This repository contains the source code for **Tempus Obice**, an interactive fantasy world atlas.

The application allows users to explore a handcrafted fantasy world through an interactive image-based map. Users can click locations on the map to learn about them and navigate to stories written in Markdown.

The project is intended to be highly maintainable, scalable, and data-driven. New locations and stories should be addable without modifying application logic.

---

# Primary Goal

Build an interactive world map where:

- A large fantasy map image is displayed.
- Users can zoom and pan.
- Users can click locations.
- A sidebar displays information about the selected location.
- Stories are stored as Markdown.
- Story pages are statically generated by Astro.
- The project remains fast, simple, and easy to maintain.

# Core Architecture

Astro is responsible for:

- Routing
- Static rendering
- Story pages
- Content collections

React is responsible ONLY for:

- Interactive map
- Sidebar
- Client-side UI state

Leaflet is responsible ONLY for:

- Image rendering
- Zoom
- Pan
- Marker rendering

The React application should exist as a single Astro Island.

Structure:

Astro Page

↓

React App

↓

WorldMap
Sidebar

Only App owns shared state.

---

# State Management

There is exactly one source of truth.

selectedLocation

This state is stored inside App.jsx.

WorldMap updates it.

Sidebar reads it.

No component besides App should own application state.

Do NOT introduce Context API.

Do NOT introduce Redux.

Do NOT introduce Zustand.

# Data Model

## Location

Represents one clickable location.

Fields:

name

description

coordinates

markerIcon

storyIds

Coordinates are stored in image pixel space.

They are NOT latitude and longitude.

---

## Story

Represents one Markdown document.

Fields:

title

locationId

summary

order

Markdown body

Stories belong to one location.

A location can have many stories.

## Leaflet Architecture Rules

The Leaflet map instance must have a single owner.

Future features (markers, flyTo, overlays, highlighting) should extend the existing map rather than recreate it.

Do not recreate the Leaflet instance during React re-renders.

Prefer updating layers over rebuilding the map.

Map initialization and marker rendering should remain separate responsibilities.
